\section{Testing}

The project includes unit-testing, which provides a way to assess functionality of individuals parts of the codebase, and integration-testing, which judges the result of the cooperation of the individual parts.

\subsection{Unit Test}

A good portion of the code has complete coverage with unit-tests: almost 100\% of the business logic and DAOs. Mocking the DAOs opens the possibility to explore every branch of the endpoints, and providing access to the underline store allows for complete access during InMemDAOs evaluation.

We used package private field, in combination with a symmetric folder structure, to allow us to access fields and methods for testing. For example the \verb|CryptoService| class exposes as package-private the \verb|setClock(Clock)| method, which allows us to "move" the clock as we please to test for expiration of tokens. This method is not accessible otherwise.

The models are not taken directly into consideration, as they are all POJOs, and are still collaterally covered nearly to 100\% by unit and integration tests.

\subsection{Integration Test}

This architecture makes integration testing cumbersome, as it requires spinning up a lightweight HTTP server just to handle test requests. One of the problem with this, is that it ends up as being mostly a test for the library implementation and not for the application code itself; that's why we decided to perform most of this tests using industrial-level application like Insomnia, which is an application designed to test HTTP endpoints.

Instead, we opted for automated integration testing for the PostgreDAOs, which require a database to connect to.

\paragraph{PostgreDAOs testing} 

The postgre\_dao package is tested through integration tests that interact with a real database instance. To ensure that the tests run in an isolated and reproducible environment, the project uses Testcontainers which spins up a PostgreSQL container.

A base test class initializes a PostgreSQL container. The container is configured with a test database name, username, and password. Additionally, the initialization SQL script (init.sql) is executed to create the required database schema before the tests run.

Each DAO test class extends this base test class, allowing it to reuse the same container configuration. After each test execution, the database tables are cleared using SQL TRUNCATE and then repopulated with a fixed set of data. This guarantees that every test does not interfere with other tests.

\vspace{1em}

\begin{lstlisting}[language=Java, caption=PostgreIT class responsible for continer initialization, label=PostgreIT]
	public abstract class PostgreIT {
		static final PostgreSQLContainer<?> postgres;
		static {
			postgres = new PostgreSQLContainer<>("postgres:16")
			.withDatabaseName("testdb")
			.withUsername("test")
			.withPassword("test")
			.withInitScript("init.sql");
			postgres.start();
			
			DatabaseManager.initWithCredentials(
			postgres.getJdbcUrl(),
			postgres.getUsername(),
			postgres.getPassword()
			);
		}
		@AfterEach
		void cleanUp() throws SQLException {
			try (var stmt = DatabaseManager.getInstance().getConnection().createStatement()) {
				stmt.execute("TRUNCATE Users, Admin, Pin_Challenge, School, Professor, Degree, Course, Teaching," +
				"Drafted_Review, Published_Review, Up_vote, Report RESTART IDENTITY CASCADE");
			}
		}
	}
\end{lstlisting}
